## ADDED Requirements

### Requirement: Grounded RAG answer endpoint

The system SHALL expose `GET /api/courses/{course_id}/ask?q=<question>` that returns a JSON object `{ query, answer, sources, cached }`, where `answer` is generated by the LLM using ONLY the top note chunks retrieved for the question (via the course-scoped retrieval and the one embedding model), and `sources` is the list of those chunks. The answer SHALL NOT introduce facts absent from the retrieved notes, and SHALL state when the notes do not cover the question.

#### Scenario: Question answered from notes

- **WHEN** an authenticated owner asks a question for which the course has relevant notes
- **THEN** the system SHALL retrieve the top chunks, generate an answer grounded only in them, and respond with `{ query, answer, sources, cached: false }`

#### Scenario: No relevant notes

- **WHEN** retrieval returns no chunks for the question
- **THEN** the system SHALL respond with an answer indicating the notes don't cover that, `sources` empty, and `cached: false`, without calling the LLM

#### Scenario: Blank question

- **WHEN** the `q` parameter is empty or whitespace
- **THEN** the system SHALL return an empty answer with `sources` empty and `cached: false`

### Requirement: Owner-scoped and gated Q&A access

The system SHALL require an authenticated user for the ask endpoint, scope it to a course the caller owns (returning HTTP 404, never 403, for a course the caller does not own or that does not exist), and return HTTP 503 with a `qa_disabled` error code when the `enable_qa` setting is false.

#### Scenario: Non-owner is denied as not-found

- **WHEN** a caller asks against a course owned by another user
- **THEN** the system SHALL respond HTTP 404 with a `course_not_found` error and SHALL NOT reach the answer logic

#### Scenario: Q&A disabled

- **WHEN** `enable_qa` is false
- **THEN** the system SHALL respond HTTP 503 with error code `qa_disabled`

### Requirement: Semantic cache serves near-duplicate questions

The system SHALL maintain a per-course semantic cache of recent answers keyed by question embedding. On an ask, after embedding the question, the system SHALL return a cached answer (with `cached: true`) when a non-expired cached entry for that course has cosine similarity to the question at or above `cache_similarity_threshold`, without performing retrieval or an LLM call. On a cache miss, the system SHALL compute the answer and store it for future lookups.

#### Scenario: Near-duplicate question hits the cache

- **WHEN** a question is asked whose embedding is within `cache_similarity_threshold` of a non-expired cached question in the same course
- **THEN** the system SHALL return the stored answer and sources with `cached: true`, skipping retrieval and the LLM

#### Scenario: Dissimilar question misses

- **WHEN** no cached entry for the course meets the similarity threshold
- **THEN** the system SHALL compute a fresh answer (`cached: false`) and add it to the course's cache

#### Scenario: Cache is course-scoped

- **WHEN** a cached entry exists only under a different course
- **THEN** it SHALL NOT be returned for this course's question

### Requirement: Cache freshness via expiry and invalidation

The system SHALL bound cache staleness three ways: each entry SHALL be ignored once older than `semantic_cache_ttl`; the per-course cache key SHALL carry that TTL as a backstop; and the system SHALL invalidate (clear) a course's cache when a new lecture for that course becomes `ready`. The per-course cache SHALL be bounded to `semantic_cache_max_per_course` most-recent entries.

#### Scenario: Expired entry ignored

- **WHEN** the only matching cached entry is older than `semantic_cache_ttl`
- **THEN** the system SHALL treat the lookup as a miss and recompute

#### Scenario: New lecture clears the cache

- **WHEN** a lecture pipeline completes and sets a lecture to `ready`
- **THEN** the system SHALL invalidate that course's semantic cache so new content is not masked by stale answers

#### Scenario: Cache size bounded

- **WHEN** stored entries for a course exceed `semantic_cache_max_per_course`
- **THEN** the system SHALL retain only the most recent entries up to that bound

### Requirement: Cache degrades gracefully without Redis

The system SHALL treat the semantic cache as optional: when `redis_url` is not configured or Redis is unreachable, cache lookups SHALL miss and cache stores/invalidations SHALL be no-ops, so the ask endpoint continues to answer by computing every time.

#### Scenario: No Redis configured

- **WHEN** `redis_url` is blank
- **THEN** every ask SHALL be computed fresh (`cached: false`) and no cache operation SHALL raise

#### Scenario: Redis error during a cache operation

- **WHEN** a Redis call fails during lookup, store, or invalidate
- **THEN** the system SHALL swallow the error and proceed as a cache miss / no-op rather than failing the request
